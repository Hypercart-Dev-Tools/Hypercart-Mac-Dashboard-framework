## How it fits together
The **DashboardDemoStore** serves as the central state manager, coordinating navigation (`DashboardPresentationStore`), load states, and maintenance operations via `DashboardOperationStore`. It injects the `DashboardRuntimeEnvironment` for logging and system services, enabling cross-cutting concerns like notifications and file access.  

**DashboardDemoModels** defines the data structures (e.g., `DashboardDemoItem`) used throughout the app, providing typed representations of UI elements with status and metadata. These models are consumed by the store and UI layers to render dynamic content.  

The **DashboardRuntimeEnvironment** acts as a dependency container, wiring together logging, file management, and user defaults. It’s injected into the store and operations module, ensuring consistent access to system services across layers.  

**DashboardOperationStore** and related types (`DashboardOperationPhase`, `DashboardOperationStep`) model asynchronous tasks, tracking progress through phases and steps. These operations are orchestrated by the store, enabling maintenance actions like data syncing or status checks.  

The **HypercartMacOSDashboard** umbrella target aggregates core logic and UI, exporting key modules for external use while keeping the demo app’s state management and environment setup tightly coupled.
